## Problem Statement

RoboWBC v0.1.0 has a solid architecture and the GEAR-SONIC planner velocity path already works with real ONNX models. What is still missing: (1) the real encoder+decoder tracking contract for the full 3-model pipeline, (2) a working benchmark that exercises real model inference, and (3) real benchmark numbers in docs. Without real model inference, the project cannot prove its core value proposition: a unified, config-driven inference runtime for humanoid whole-body control policies.

## What Makes This Cool

The "whoa" moment is running NVIDIA's production GEAR-SONIC model — three ONNX files, 1.2M–42M parameters, validated on real Unitree G1 hardware — entirely inside a Rust runtime. If robowbc can match or beat NVIDIA's C++ deployment latency, it proves Rust is viable for real-time humanoid control. That gets robotics researchers and the Rust community excited.

## Constraints

- Must use the existing \`robowbc-ort\` ONNX Runtime backend (ort crate)
- Must preserve the existing \`WbcPolicy\` trait — no architecture rewrites
- Must run at 50 Hz control frequency (industry standard for WBC)
- Must work on CPU first; CUDA/TensorRT EP is a bonus
- Must integrate with existing MuJoCo sim transport for hardware-free validation

## Premises

1. The \`WbcPolicy\` trait + \`WbcCommand\` enum is the right level of abstraction — real GEAR-SONIC inputs (motion tokens, proprioception, IMU) map cleanly to \`Observation\` and \`WbcCommand::MotionTokens\`.
2. Real ONNX models are the next step because they validate both the architecture and the performance claims. Mock fixtures cannot prove latency or correctness.
3. Python bindings should wait until the core Rust API is validated with real models. A stable trait boundary makes bindings durable.

## Approaches Considered

### Approach A: Real GEAR-SONIC End-to-End
Download actual NVIDIA \`GEAR-SONIC\` ONNX weights from HuggingFace, wire the 3-model pipeline (encoder → planner → decoder) with correct tensor shapes, and run inference end-to-end in MuJoCo sim. Update benchmarks with real numbers. This is the highest-leverage move — it validates the architecture, fills the benchmark table, and creates a demo to show researchers.

- **Effort**: M
- **Risk**: Med
- **Pros**: Proves runtime works; unlocks community demos; validates trait API; benchmark data for content
- **Cons**: Requires understanding NVIDIA's exact tensor shapes and ZMQ protocol v4; may need model-specific tweaks
- **Reuses**: Existing \`robowbc-ort\`, \`robowbc-sim\`, \`robowbc-cli\`, benchmark framework

### Approach B: Bindings + LeRobot Bridge First
Ship PyO3 Python bindings and a LeRobot backend adapter early. This maximizes distribution reach but risks binding an unstable Rust API before real models prove the trait boundary.

- **Effort**: M
- **Risk**: Med-High
- **Pros**: Reaches Python-first robotics community; potential LeRobot integration
- **Cons**: API may shift during real model integration; bindings become tech debt
- **Reuses**: Existing crate structure

### Approach C: Hardware Transport First
Implement Unitree G1 hardware transport via CycloneDDS or zenoh-ros2dds bridge. Proves sim-to-real capability but without real models, the transport is just a pipe with no payload.

- **Effort**: M-L
- **Risk**: Med
- **Pros**: Real robot demo; safety layer design
- **Cons**: No value without working policy inference; harder to debug without sim
- **Reuses**: Existing \`robowbc-comm\` zenoh layer

## Recommended Approach

**Approach A: Real GEAR-SONIC End-to-End**

Rationale: This is the one thing that unlocks everything else. Real ONNX inference proves the runtime works, validates the \`WbcPolicy\` trait API, generates benchmark numbers for the "Why Every Humanoid Team Rebuilds WBC Deployment from Scratch" article, and creates a demo worth sharing with the NVIDIA and LeRobot communities. Hardware transport (C) and Python bindings (B) both become easier once the core inference path is solid.

## Open Questions

1. What are the exact input/output tensor shapes for \`encoder_sonic.onnx\`, \`decoder_sonic.onnx\`, and \`planner_sonic.onnx\`? The founding doc references them but the code may need updates.
2. Does the ZMQ streaming protocol v4 (1280-byte headers) matter for robowbc, or is that only in NVIDIA's C++ deployment stack? If robowbc loads models directly via ONNX Runtime, the ZMQ protocol may be irrelevant.
3. Can \`ort\'s CPU EP run the full 3-model pipeline within the 20ms tick budget (50 Hz)? If not, what's the bottleneck — session creation, inference latency, or tensor marshalling?

## CI Strategy for Ignored Integration Test

The real-model integration test is marked `#[ignore]` to avoid CI failures on runners without GPU or large model downloads. To prevent bit-rot:
- Run the ignored test locally before each release
- Add a scheduled CI job (weekly) on a self-hosted runner with model cache to execute ignored tests

## ZMQ Footprint in Codebase

No ZMQ code currently exists in robowbc. The ZMQ protocol v4 mentioned in the founding doc is part of NVIDIA's C++ `gear_sonic_deploy` stack. Since robowbc loads ONNX files directly via `ort`, ZMQ interop is only relevant if we choose to bridge to NVIDIA's deployment binary — not required for this approach.

## Parameter Breakdown

Per the founding doc and HuggingFace model card (`nvidia/GEAR-SONIC`):
- `model_encoder.onnx`: encodes motion reference into latent tokens
- `model_decoder.onnx`: decodes latent into joint targets
- `planner_sonic.onnx`: real-time locomotion planner

Exact parameter counts per file should be verified from the HuggingFace model card or ONNX metadata after download. The 1.2M–42M range reflects the full GEAR-SONIC family (smallest to largest checkpoint).
