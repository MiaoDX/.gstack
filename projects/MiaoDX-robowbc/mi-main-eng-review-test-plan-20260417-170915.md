- `robowbc run --config configs/sonic_g1.toml` — validate the default velocity path still works and the emitted report metadata stays truthful.

## Key Interactions to Verify

- Run `configs/sonic_g1.toml` after downloading models and confirm the chosen command wording matches CLI behavior, JSON report output, and showcase metadata.

- Generate a JSON report and confirm `command_kind`, `command_data`, and `report_version` are accepted by showcase/report consumers.
- Run renamed `gear_sonic` benchmarks and confirm docs distinguish cold-start, warm steady-state, replan, tracking, and end-to-end numbers.

## Edge Cases
- Conflicting runtime fields in one config fail fast with a clear error.
- Empty `motion_tokens` is either explicitly rejected or explicitly mapped to the chosen Gear-Sonic alias, matching docs.
- Python SDK docs and `examples/python/gear_sonic_inference.py` use the correct 29-DOF G1 expectations and current command semantics.
- `robowbc init` template comments do not regress to fixture-only or stale tracking language.
- Public community/showcase docs no longer claim a generic "full 3-model pipeline" where none exists.

## Critical Paths
- Fresh contributor path: clone -> `cargo build` -> `cargo run --bin robowbc -- run --config configs/decoupled_smoke.toml`
- Live GEAR-Sonic path: download models -> `cargo run --release --bin robowbc -- run --config configs/sonic_g1.toml` -> report/benchmark docs agree on what ran
- Benchmark truth path: `GEAR_SONIC_MODEL_DIR=models/gear-sonic cargo bench -p robowbc-ort -- gear_sonic` -> published tables and roadmap language match measured modes
- Artifact compatibility path: generate JSON/showcase report -> downstream consumers render the chosen command labels correctly
