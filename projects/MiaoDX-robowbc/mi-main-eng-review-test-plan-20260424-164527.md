## Affected Pages/Routes
- `/policies/gear_sonic/` — verify the published phase-first detail page loads with the documented default phase, moment, lag, and camera.
- `/policies/decoupled_wbc/` — verify the same contract works for another velocity-schedule showcase and preserves truthful lag availability.
- `/policies/wbc_agile/` — verify the reusable contract is not hardcoded to one policy family.
- `/policies/gear_sonic_tracking/` — verify generic checkpoint rendering without a sidecar and phase-aware rendering when a valid sibling sidecar exists.

## Key Interactions to Verify
- Selecting a phase changes the primary evidence stage and keeps `track` as the primary camera.
- Switching from phase end to midpoint hides or disables lag selection with clear helper text.
- Lag buttons expose only the actual available `+0..+5` options for the selected phase-end checkpoint.
- Generic MuJoCo proof packs without phase-review capability stay on the current checkpoint flow.
- Capability-enabled proof packs render `phase-timeline`, `phase-lag-selector`, and diagnostics without regressing the local `run.rrd` embed path.

## Edge Cases
- Partial or duplicate `phase_name` usage in a velocity schedule fails validation.
- Fractional segment durations round deterministically into `start_tick`, `midpoint_tick`, and `end_tick`.
- Final-phase tail preservation keeps the published showcase honest for `+0..+5` lag review.
- Tracking sidecars with overlapping, unsorted, or out-of-range ranges fail loudly instead of degrading to generic behavior.
- Manifest labels and relative paths reject HTML-injection strings, absolute paths, and `..` escapes.

## Critical Paths
- `runtime.velocity_schedule` authoring -> CLI `phase_timeline` -> replay trace -> proof-pack manifest -> static detail page -> bundle validator.
- Generic tracking run -> report script generic checkpoint flow -> static detail page remains generic.
- Tracking run with valid sibling `.phases.toml` -> capability-enabled manifest -> phase-aware detail page -> validator requires lag assets.
- Targeted Python contract tests -> `make showcase-verify` -> open the generated policy detail page and confirm the documented default state matches the manifest.
