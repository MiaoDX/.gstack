## Problem Statement

LIP should become a trusted public practice archive for AI Coding, OpenClaw, and agent engineering. The target reader is an AI-curious robotics/backend engineer in a Chinese tech company who asks: "Am I still near the frontier of making AI actually useful in my daily work?"

The active asset is `ai-coding/ultrathink-to-goal/`. It should not only be a 50-minute talk deck. It should become the first reusable internal-showcase artifact: a serious engineer can read it, trust the proof, hand one part to Claude/Codex, and try a workflow at work tomorrow.

## What Makes This Cool

The useful version of LIP is not another AI news site. It is a field lab:

- Real practice, not demo-only AI coding claims.
- Hardcore daily engineering workflows, especially robotics/backend-adjacent work.
- Open-source/MIT-friendly artifacts people can reuse.
- Articles that include enough source, prompt, expected output, and verification detail for readers to reproduce.
- Talks that convert into internal showcases and lightweight consulting opportunities.

The "whoa" moment is: "I can test this on my job tomorrow, or now."

## Constraints

- Keep it simple. Do not redesign the deck or build a full lab system now.
- `ai-coding/ultrathink-to-goal/index.html` remains the canonical deck.
- Avoid turning open practice content into a hard paid-wall funnel.
- Consulting CTA should stay light because `miaodx.com` already carries the broader consulting/talk profile.
- Any new source artifacts must follow repo rules: AI Coding project material lives under `ai-coding/<slug>/`, with local assets under that slug.
- The source should remain useful to readers even if they never hire MiaoDX.

## Premises

1. LIP's reader is an AI-curious robotics/backend engineer in a Chinese tech company who wants a practical frontier check.
2. LIP's strongest article format is not essay-only. It is article + proof + source + agent handoff + verification path.
3. `ai-coding/ultrathink-to-goal/` should act as the first productized 90-minute internal showcase, not only a one-off deck.
4. Monetization should start with paid internal showcases/workshops, not a paid website/community.
5. The 24-hour user action is: try one demo from the talk on real work and see a visible result.
6. The first implementation should stay KISS: one companion practice page, one README link, one deck callout.

## Approaches Considered

### Approach A: Deck-First Showcase Pack

Update the deck itself so the live talk drives people toward three "try tomorrow" demos.

Pros:

- Strong live-talk conversion.
- Audience sees the call to action during the presentation.

Cons:

- Risks bloating a talk that already has a tight narrative.
- Harder to maintain detailed reproduction steps inside HTML slides.

### Approach B: Companion Practice Pack

Keep the deck focused. Add a simple companion page beside it with copy-paste demos and verification steps.

Pros:

- Best maintainability.
- Serious engineers can hand the page directly to Claude/Codex.
- Keeps the deck from becoming a manual.

Cons:

- Weaker live-talk action moment unless the deck links to it.

### Approach C: KISS Hybrid

Add one companion `practice.md`, one README section linking it, and one lightweight deck link/QR on the final Tips or Q&A slide.

Pros:

- Smallest complete version of the showcase pack.
- Supports both live audience action and after-talk reproduction.
- Reuses the existing deck and project directory without inventing a new content system.

Cons:

- Does not yet create a site-wide reusable "practice lab" framework.
- The first version will prove the format manually before automation or templates exist.

## Recommended Approach

Choose Approach C: KISS Hybrid.

Desired output:

1. Add `ai-coding/ultrathink-to-goal/practice.md`.
2. Update `ai-coding/ultrathink-to-goal/README.md` with one "Try this tomorrow" section.
3. Update `ai-coding/ultrathink-to-goal/index.html` only with a final-slide link/QR callout to the practice page.

Do not redesign the deck. Do not add a full consulting page here. Do not build a generic LIP practice framework yet.

## Practice Page Shape

`practice.md` should have the title: "Try This Tomorrow".

Keep it to three demos max. Each demo should include:

- Goal
- When to use it
- Copy-paste agent prompt
- Required repo/file context
- Expected result
- Verify step
- Failure mode or caveat

End with:

- Open-source reuse note: source material is intended to be reused under the repo license unless a linked external source says otherwise.
- Contact note: for internal team showcase/setup, use the contact path on `miaodx.com`.

## Candidate Demos

These are placeholders for the implementation pass to refine:

1. Ask your agent to interview you before implementation.
   - Goal: prevent vague tasks from turning into wrong code.
   - Verify: the agent produces clarified acceptance criteria before editing.

2. Run one real issue through a goal-oriented agent workflow.
   - Goal: compare "prompt and pray" with a plan/verify loop.
   - Verify: the output includes changed files, test or screenshot evidence, and remaining risks.

3. Turn one piece of team practice into agent-readable repo context.
   - Goal: move tacit workflow from chat/head into versioned docs.
   - Verify: a new agent session can find the rule from the repo without being told verbally.

## Success Criteria

- A reader can pick one demo and run it within 24 hours.
- Each demo has a copy-paste prompt and a verification step.
- The deck remains coherent as a talk.
- The README clearly exposes the practice path.
- The artifact supports a 90-minute internal showcase without requiring custom prep every time.
- At least three future readers/audience members can report which demo they tried and whether it worked.
