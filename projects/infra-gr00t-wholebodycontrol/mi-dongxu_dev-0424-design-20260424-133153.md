## Problem Statement

The repo already supports two object families, cylinders and spheres, and two
robot families, G1 and Realman. The current implementation works well enough
to pass the representative harness envelope, but it is not a good foundation
for extending grasp support to more objects.

The core problem is not just the two operator-visible weird-wrist cases. Those
cases are the warning sign that object semantics and robot execution semantics
are still too entangled. If that layering stays blurry, every new object family
will risk growing fresh G1-only and Realman-only exceptions.

The desired end state for the current supported object families is:

- object-specific differences live mostly in `grasp point`, `grasp family`,
  and `entry line`
- robot-specific differences live mostly in reachability, waist motion, and
  unavoidable execution seams
- the shared flow stays as similar as possible across robots:
  `pregrasp -> approach -> close -> lift`
- the public API remains `plan_reach_to_grasp()`
- the weird-wrist cases visibly improve rather than merely becoming better
  explained

## What Makes This Cool

This turns the current grasp code from a collection of robot-shaped behaviors
into a cleaner semantic core.

That matters because the repo is not trying to support "everything" yet. It is
trying to make the existing `cylinder + sphere` support clean enough that the
next object family does not require another round of robot-specific repair
work.

The coolest version of this is simple:

- one shared grasp mental model
- two robot adapters with explicit, honest differences
- fewer hidden semantic drifts
- better posture on the cases humans already notice

## Constraints

- Keep the public planner entrypoint `plan_reach_to_grasp()`.
- Keep the canonical runtime command contract on the repo-owned
  `ControlGoalPacket`.
- Keep the supported G1 backend boundary from Phase 68 fixed:
  repo-owned wrapper over official `decoupled_wbc`, SONIC experimental.
- Do not reopen the already-regressive broad fallback removal or broad
  probe-hint widening experiments as defaults.
- Preserve the dual-robot representative 8-case visual harness as the
  no-regression gate.
- Preserve timing envelopes as a tracked acceptance surface, not just pass/fail.
- Allow G1 to retain irreducible special handling where waist motion is
  genuinely required to reach or execute the grasp.

## Premises

1. The main Phase 67 problem is shared semantic cleanup for the existing
   `cylinder + sphere` object support, not just isolated weird-wrist
   attribution.
2. Weird-wrist remains an important acceptance signal because the user does not
   consider "same bad visible posture, better explanation" a success.
3. The right split is object contract first, robot execution second.
4. "Same API, explicit semantics" is better than either:
   a) fake identical behavior behind one API, or
   b) growing separate public APIs per robot.
5. A full object-planner / robot-executor split is desirable, but it is too
   large to hide inside the current Phase 67 cleanup slice.

## Approaches Considered

### Approach A: Local Weird-Wrist Repair Only

Treat Phase 67 as a narrow G1 behavior patch. Improve the suspicious cases with
selector tweaks or close-in posture tweaks, document what remains, and stop.

- Effort: Small
- Risk: Low
- Pros:
  - fastest path to local visible improvement
  - least likely to disturb the existing harness envelope
  - fits the current `67-02` draft with minimal change
- Cons:
  - does not materially improve extensibility
  - leaves object and robot semantics mixed together
  - likely re-creates the same architecture argument when a new object family is added
- Reuses:
  - current `67-02` diagnostic framing
  - `curobo_grasp_planner.py`
  - current harness and `approach_endpoint_*` metrics

### Approach B: Shared Semantic Contract Cleanup For Existing Object Families

Keep one public API, but tighten one shared internal semantic contract for the
current `cylinder + sphere` families. The object layer decides the grasp
geometry intent. The robot layer decides how to execute it honestly.

- Effort: Medium
- Risk: Medium
- Pros:
  - directly reduces future maintenance cost
  - gives one cleaner place to add future object families
  - lets weird-wrist improvement be part of semantic convergence, not a one-off
  - keeps Phase 67 bounded
- Cons:
  - larger than a pure selector patch
  - may expose some current G1 behavior as explicitly non-shareable
  - requires disciplined boundaries to avoid leaking robot-specific quirks back
    into the shared object layer
- Reuses:
  - `plan_reach_to_grasp()`
  - generalized request/result contract from Phase 65
  - current `GraspPlan` / `GraspStage` flow
  - shared `approach_endpoint_*` diagnostics

### Approach C: Full Object Planner / Robot Executor Split

Compile an object-centric grasp recipe first, then execute it through robot
adapters. G1 and Realman become execution backends over one semantic core.

- Effort: Large
- Risk: High
- Pros:
  - best long-term architecture
  - cleanest extensibility story for new object families
  - strongest defense against silent semantic drift
- Cons:
  - too large for the current cleanup slice
  - highest regression risk
  - should be its own follow-on phase, not hidden inside Phase 67
- Reuses:
  - public request contract
  - stage-driven runtime
  - harness and docs as regression oracle

## Recommended Approach

Choose **Approach B** for Phase 67.

This is the smallest thing that solves the actual problem. It is large enough
to improve extensibility for the currently supported object families, but still
small enough to stay inside the existing dual-robot harness and timing gate.

The key design rule is:

- object layer owns:
  `grasp point`, `grasp family`, `entry line`, and the intended
  `pregrasp -> approach -> close -> lift` contract
- robot layer owns:
  reachability, waist dependency, and any unavoidable execution-time mutation

The public API stays shared. Remaining behavior differences become explicit
semantic markers rather than hidden drift. Recommended markers:

- `approach_contract_mode`
- `close_in_acceptance_mode`
- `waist_dependency_mode`
- `runtime_stage_mutation`

The design target is not "make G1 pretend it is Realman." The design target is
"make both robots share the same grasp semantics unless a real robot
constraint forces a difference."

### Phase 67 Scope Under Approach B

Phase 67 should be reframed as:

**Goal:** clean up the shared semantic contract for the currently supported
`cylinder + sphere` grasp families so object-driven logic stays shared and
robot-driven execution differences stay bounded and explicit.

That reframing changes the role of the current `67-02` work:

- `early winner selection` becomes one candidate mechanism to test
- it is no longer the center of the phase
- weird-wrist improvement remains an important acceptance outcome
- any change must still hold the dual-robot representative harness and timing
  envelope

### Proposed New Follow-On Phase For Approach C

Add a new future phase after Phase 68:

**Phase 69: Split Object Grasp Semantics From Robot Execution Adapters**

Proposed goal:

- compile one object-centric grasp recipe for the supported families
- make G1 and Realman explicit execution adapters over that recipe
- move remaining robot-specific logic behind stable adapter seams instead of
  letting it leak into the object/grasp semantic layer

Phase 69 should only start after Phase 67 proves the smaller shared-contract
cleanup is safe under the current dual-robot regression gate.

## Open Questions

- Where should the shared object-centric semantic contract live:
  inside `curobo_grasp_planner.py` first, or in a smaller dedicated helper/module?
- Should `sphere` and `cylinder` share one normalized family schema with
  different `grasp_family_id` values, or should sphere-specific top-entry stay
  intentionally separate at the semantic-contract level?
- Which current G1 behaviors are truly irreducible robot constraints, and which
  are merely historical implementation artifacts?
- What is the smallest measurable "visible improvement" threshold for the
  weird-wrist cases so the phase does not hide behind pass/fail only?

## Success Criteria

Phase 67 is successful if all of the following are true:

- The repo retains one public grasp API: `plan_reach_to_grasp()`.
- The shared semantic contract for `cylinder + sphere` is clearer and more
  centralized than today.
- Object differences are mostly represented through grasp geometry intent,
  not robot-specific flow branching.
- G1-only special behavior is reduced to real reachability/waist/runtime needs.
- The weird-wrist cases improve visibly enough that the user does not describe
  them as "still weird."
- G1 still holds at least `6 pass / 2 fail / 0 execution_error` on the
  representative 8-case harness.
- Realman still holds at least `7 pass / 1 fail / 0 execution_error` on the
  representative 8-case harness.
- Tracked timing metrics do not materially regress against refreshed baselines.

  differences can exist in grasp point selection, but "其他流程尽可能一样".
