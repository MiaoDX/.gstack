## Problem Statement

The current grasp stack is too bottle-shaped.

The public and internal planner contracts are still heavily influenced by:

- bottle yaw sampling,
- side-entry bottle assumptions,
- explicit pregrasp and approach stage logic,
- object-specific branching that makes new grasp families feel like new planners.

This blocks two things the team now needs:

1. a cleaner cuRobo v2-backed grasp API that callers can actually tweak and extend, and
2. a planner family that can naturally cover:
   - current single-hand side-entry bottle grasps,
   - current top-down grasps,
   - future two-hand box grasps.

The key migration goal is **not** "force everything into one optimizer call."
The real goal is **one clean public grasp contract** that can hide internal
sub-solves when needed, while removing bottle-only assumptions from the API.

## What Makes This Cool

The exciting version of this migration is not just "new cuRobo version."

It is this:

- one generic grasp request type,
- one public `reach_to_grasp` style planner surface,
- side-entry and top-down as presets instead of separate planner branches,
- multi-hand-capable request shape from day one,
- bottle as the first benchmark case, not the abstraction.

If this works, the planner stops feeling like "the bottle planner" and starts
feeling like a real grasp-intent engine.

## Constraints

- The first backend must use cuRobo v2. v1 is being deprecated internally.
- The first implementation wedge should stay single-arm in execution, but the
  request type must already be able to represent one hand or two hands.
- The first demo should target the current no-waist upright bottle-on-table case.
- Internal sub-solves are allowed. The first version does **not** have to prove
  a true single continuous optimization from start to close.
- Human-like motion shaping is desirable, but not a first-version API promise.
  If cuRobo v2 makes style shaping easy later, that can become a follow-on.
- The first version must naturally support both:
  - side-entry grasp presets,
  - top-down grasp presets.
- The new API must not encode bottle-only nouns or bottle-only geometry choices
  into the public request shape.

## Premises

1. The biggest current problem is not raw grasp quality, it is that the planner
   contract is overfit to bottle logic and therefore hard to extend.
2. The first migration should optimize for a cleaner public API, not for "one
   continuous solve at all costs."
3. The right first wedge is a single-arm generalized path whose request type is
   already multi-hand-capable, even though the first implementation executes the
   bottle benchmark only.

## Approaches Considered

### Approach A: Thin v2 Wrapper, Fastest Ship

Add a new generic request type, but mostly map it onto the current staged logic.
Bottle side-entry and top-down become presets, but the internal planner shape
remains close to today.

- Effort: Medium
- Risk: Low to Medium
- Pros:
  - fastest path off cuRobo v1
  - quickest demo of a cleaner public call
  - preserves most of the current execution pipeline
- Cons:
  - keeps too much bottle-era planner shape alive
  - box support likely still feels bolted on
  - phase complexity shrinks only cosmetically
- Reuses:
  - current stage execution
  - current attachment/world handling lessons
  - current grasp-close runtime wiring

### Approach B: Generic Request + v2 Intent Compiler

Build a new generic `GraspRequest` and compile it into cuRobo v2 solve inputs.
Side-entry bottle and top-down are presets built on top of the same request
shape. Internal sub-solves are allowed, but hidden behind one public call.

- Effort: Large
- Risk: Medium
- Pros:
  - clean public API
  - bottle no longer defines the planner shape
  - top-down support fits naturally
  - future box path has a real slot instead of an afterthought
  - makes the planner obviously easier to extend
- Cons:
  - more design work up front
  - requires a new cuRobo v2 adapter layer
  - first benchmark validation matters a lot
- Reuses:
  - existing execution path
  - existing runtime world/attachment lessons
  - current benchmark scenes and harnesses

### Approach C: Full Generalized Manipulation Planner Now

Do the whole migration in one shot: generic request, multi-hand execution,
preset library, scene owner, attachment owner, bottle and box support, all on a
fully v2-native planner stack.

- Effort: Extra Large
- Risk: High
- Pros:
  - most future-proof
  - one grand migration
- Cons:
  - too many moving parts at once
  - easy to lose the wedge
  - hard to prove value early
- Reuses:
  - mostly ideas rather than code shape

## Recommended Approach

Choose **Approach B: Generic Request + v2 Intent Compiler**.

This is the smallest plan that actually solves the real problem.

Approach A gets a nicer wrapper, but leaves too much old bottle logic inside the
planner core. Approach C is the ocean. Approach B is the lake:

- small enough to wedge,
- big enough to fix the abstraction,
- disciplined enough to keep bottle from becoming the API again.

### Recommended architecture

#### 1. New public request type

Introduce a repo-local `GraspRequest` shape with concepts like:

- `effectors`: one or more hands, for example `[left]` now, `[left, right]` later
- `target_object`: abstract object reference or geometry context
- `grasp_family`: generic preset selection, not object-specific planner branch
- `entry_preference`: side-entry, top-down, or custom
- `goal_candidates`: optional externally supplied candidate poses later
- `scene_context`: world information needed for planning

The public type should be able to represent:

- single-hand bottle side-entry,
- single-hand bottle top-down,
- future two-hand box grasps,

without changing the request shape.

#### 2. Presets compile into planner intent

Do **not** expose bottle yaw sampling as a public concept.

Instead:

- `side_entry` preset compiles into candidate generation / entry constraints
- `top_down` preset compiles into a different candidate generation / entry intent
- future `bimanual_box` preset compiles into paired end-effector goals

This keeps object-specific heuristics behind the compiler boundary.

#### 3. cuRobo v2 adapter layer

Add a dedicated v2 adapter rather than mutating the current v1 wrapper in place.

The adapter owns:

- `GoalToolPose`-style candidate construction,
- v2 `MotionPlannerCfg` setup,
- capacity planning for `max_goalset` / `max_batch_size`,
- scene translation into `SceneCfg`,
- result normalization into repo-local planner results.

This is the right place to translate request intent into v2 planner primitives.

#### 4. Internal sub-solves are allowed

The first version should not promise one optimizer call from start to grasp close.

Allowed first-version shape:

- public `plan_reach_to_grasp(request, current_state) -> GraspPlan`
- internal decomposition into goalset selection, reach prefix, close-in segment,
  optional lift preparation, if that keeps reliability high

The important rule is that callers do **not** see `pregrasp` and `approach` as
API-level planning modes anymore.

#### 5. Multi-hand-capable request, single-arm first execution

The request envelope should already allow multiple effectors, but the first
implemented execution path can support:

- one hand only,
- bottle benchmark only,
- side-entry and top-down presets.

This is the clean way to avoid another API migration when the box path lands.

#### 6. Delay motion-style knobs

Do not put "human-like," "minimal wrist," or joint preference weights into the
first public API.

Instead:

- leave comments/TODOs in the design and adapter boundary,
- note likely future extension points in cuRobo v2,
- keep first-version success focused on generality and simplification.

That keeps the first migration from turning into solver-config soup.

## Open Questions

1. Should the first v2 backend use built-in `plan_grasp(...)` as the primary
   path, or should it compile requests into lower-level pose / goalset solves
   first and keep `plan_grasp(...)` experimental?
2. What exact internal result contract should survive from the current planner?
   Example: carry/lift metadata, attached-object state, phase visualization hooks.
3. How much of the current stage machine should remain visible to downstream
   execution and logging once the public API hides pregrasp/approach?
4. What is the first acceptable representation for future two-hand box goals:
   paired candidate tool poses, or a higher-level object-frame hold contract?
5. Where should world ownership live in the v2 path so bottle/table/top-down
   cases reuse the same scene contract?

## Success Criteria

The migration is successful when all of these are true:

1. A new cuRobo v2-backed public grasp API exists with no bottle-specific nouns
   in the request shape.
2. The same request family can express:
   - bottle side-entry,
   - bottle top-down,
   - future multi-hand requests without API break.
3. The first benchmark demo works on:
   - upright bottle on table,
   - current no-waist start pose,
   - left-hand side-entry preset,
   - top-down preset on the same API family.
4. Callers do not need to manually reason about `pregrasp` vs `approach` to use
   the public planner surface.
5. The v2 path is at least competitive on latency after warmup, with a stretch
   goal of reducing end-to-end planning time versus the current bottle path.
6. The new planner surface is obviously easier to extend to the box line than
   the current codebase.

## Distribution Plan

This is a repo-internal migration, not a standalone package release.

- The first deliverable should ship as a new Python planner surface inside this
  repo, alongside the existing planner during migration.
- Existing deployment/runtime entrypoints should continue to call the old path
  until the new v2 path clears benchmark gates.
- CI should add focused coverage for:
  - request-shape tests,
  - side-entry bottle benchmark,
  - top-down benchmark,
  - adapter/result normalization,
  - regression check that no bottle-specific assumptions leak into the request API.

## Next Steps

1. Define the `GraspRequest` schema on paper first.
   Include one-hand and two-hand capability, side-entry and top-down presets, and
   explicitly ban bottle-specific fields from the public type.
2. Freeze the first benchmark set.
   At minimum:
   - bottle side-entry,
   - bottle top-down,
   - one placeholder request instance that proves the schema can represent a
     future two-hand box grasp.
3. Design the v2 adapter boundary.
   Be explicit about what is repo-local versus what is translated into cuRobo v2
   concepts like goalsets, scene config, and result objects.
4. Decide whether the first internal solve recipe uses:
   - built-in v2 `plan_grasp(...)`, or
   - compiled lower-level pose/goalset solves under one public call.
5. Only after those are frozen, implement the bottle benchmark path.
