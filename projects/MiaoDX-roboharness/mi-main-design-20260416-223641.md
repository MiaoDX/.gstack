# Design: Minimal Showcase Repo for roboharness Positioning

Generated by /office-hours on 2026-04-16
Branch: main
Repo: MiaoDX/roboharness
Status: APPROVED
Mode: Builder (open source / research project)

## Problem Statement

roboharness is a small, pip-installable library. But when users see a monorepo with many framework examples inside `examples/`, the mental model shifts to "this is a framework I have to live inside." A GR00T WBC user wants to `pip install roboharness` into their *existing* repo — not move their large codebase under roboharness.

The core repo already has a working LeRobot eval plugin (`origin/main`). The problem is not technical feasibility. It's **social proof for the pip-install workflow**.

## What Makes This Cool

A user clones `roboharness/showcase`, sees two real repos that use `roboharness` as a pip dependency, and thinks: "I can do this in my own project."

## Constraints

- Core repo must stay small and focused
- No framework bloat in core `pyproject.toml`
- Must demonstrate roboharness as a library, not a framework
- Must support local development against roboharness `main`, not just PyPI
- Must be runnable in CI without breaking when frameworks update

## Premises

1. LeRobot and GR00T WBC are the two highest-leverage integrations to showcase first.
2. A separate repo is necessary for positioning — users need to see roboharness consumed from the outside.
3. Two well-built showcases beat five half-built ones.
4. Framework owners won't upstream roboharness integration until it's more widely adopted.

## Approaches Considered

### Approach A: Full 5-framework showcase repo (original plan)
- LeRobot, GR00T, Pi0, SONIC, CaP-X
- **Rejected:** Scope too large, DX unsound, architectural debt, premature infrastructure

### Approach B: Keep all examples in core repo
- `examples/lerobot_eval_harness.py`, `examples/groot_wbc_showcase.py`
- **Rejected:** Fails the positioning test — users still see it as "living inside" roboharness

### Approach C: Minimal showcase repo with 2 frameworks + 5 design fixes (chosen)
- Exactly `lerobot-eval/` and `groot-wbc/`
- Local dev escape hatch, standard command contract, environment contract, smoke mode, clean landing README
- **Chosen:** Solves positioning without review debt

## Recommended Approach

Build `github.com/roboharness/showcase` with exactly two directories:

```
roboharness/showcase/
├── README.md
├── lerobot-eval/
│   ├── README.md
│   ├── requirements.txt
│   ├── run.sh
│   └── lerobot_eval_showcase.py
└── groot-wbc/
    ├── README.md
    ├── requirements.txt
    ├── run.sh
    └── groot_wbc_showcase.py
```

### The 5 Design Fixes

1. **Local dev mode**
   - Each `run.sh` checks for `ROBOHARNESS_SRC` env var
   - If set: `uv pip install -e "$ROBOHARNESS_SRC[demo]"` (fast editable install from source)
   - Otherwise: `uv pip install roboharness[demo]`
   - `run.sh` validates that `ROBOHARNESS_SRC/pyproject.toml` exists before installing

2. **Standard command contract**
   - Every showcase runs with `./run.sh`
   - Every showcase produces `results.json` with this schema:
     ```json
     {
       "framework": "lerobot",
       "status": "success" | "error",
       "checkpoints": ["path/to/step_000.png", ...],
       "metrics": {"mean_reward": 0.0, "success_rate": 0.0}
     }
     ```

3. **Environment contract**
   - Each showcase README lists: Python version, CUDA version, GPU requirement, expected runtime, disk space for model downloads
   - Each showcase README warns: "Create a separate virtualenv for each showcase to avoid dependency conflicts"

4. **Smoke mode**
   - Every showcase supports `SMOKE=1 ./run.sh`
   - **LeRobot smoke:** runs 1 episode, 10 steps, no checkpoint downloads (uses CartPole fallback or tiny mock policy)
   - **GR00T WBC smoke:** validates imports, instantiates dummy policy, runs 1 simulation step without loading real checkpoint weights
   - Target: <60 seconds for LeRobot, <120 seconds for GR00T WBC
   - Used for CI validation and quick setup checks

5. **Landing README matrix**
   One table at repo root showing:
   ```
   | Framework | Status | Hardware | Runtime | Last Tested |
   |-----------|--------|----------|---------|-------------|
   | LeRobot   | ✅     | GPU      | ~5 min  | v0.2.2      |
   | GR00T WBC | ✅     | GPU      | ~3 min  | v0.2.2      |
   ```

## Success Criteria

- A new user can clone the repo, create a venv, and run a showcase in under 10 minutes (after framework deps are installed)
- CI passes for both showcases on every push
- `SMOKE=1 ./run.sh` completes in <60s (LeRobot) and <120s (GR00T WBC)
- A roboharness contributor can test local changes with `ROBOHARNESS_SRC=../roboharness ./run.sh`

## Distribution Plan

- Repo lives at `github.com/roboharness/showcase` (prerequisite: `roboharness` GitHub org exists; if not, use `MiaoDX/roboharness-showcase` temporarily)
- No PyPI package — users clone and run directly
- CI via GitHub Actions with 2 independent jobs (one per showcase)
- CI runs against roboharness PyPI release by default; nightly job against `main` is deferred until the repo stabilizes
