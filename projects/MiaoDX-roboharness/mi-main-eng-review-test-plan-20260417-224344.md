## Affected Pages/Routes
- `README.md` front door — verify that the first-screen story shifts from demo gallery breadth to approval/evidence wedge clarity.
- `examples/mujoco_grasp.py --report` artifact pack — verify that a single run produces the expected machine artifacts plus the reviewed approval/evidence surface.
- `report.html` summary surface for the MuJoCo grasp wedge — verify run-level verdict, surfaced-case queue, canonical proof panel, degraded states, and unchanged-case count.

## Key Interactions to Verify
- Run the MuJoCo grasp example with `--report` and confirm the first viewport answers:
  what happened overall, which cases were surfaced, why a surfaced case is here, and
  what the next approval action is.
- Verify a mixed-outcome report where surfaced items include at least one hard fail,
  one ambiguous case, and one intended migration review case. Confirm ordering and CTA
  differences are intentional and legible.
- Verify that reviewers can inspect unchanged cases from the deeper archive without
  those cases appearing in the first-screen queue.
- Verify baseline-governance flow: old baseline remains authoritative during review,
  and any new-baseline promotion is a separate guarded step.

## Edge Cases
- Zero surfaced cases: the queue should render an explicit success state, unchanged
  count, and safe actions instead of a blank list.
- Contract invalid before execution: the review surface must block approval and explain
  problem + cause + fix.
- Partial evidence: missing current or baseline proof should remain reviewable only if
  the plan says so, with explicit banners and blocked approval where required.
- Motion-window anti-goal cannot be proven from stills: the run should surface an
  ambiguity or evidence-degraded state, not silently pass.
- Repeated failure signature stops the loop: confirm the surfaced report shows that the
  run is stuck and points to the right rerun boundary.
- Headless use: console output alone should expose verdict, surfaced count, baseline
  status, and rerun guidance without requiring a browser.

## Critical Paths
- Regression run -> compile constrained contract -> execute MuJoCo wedge -> surface only
  changed or ambiguous cases -> review against old baseline -> approve or reject run.
- Migration run -> compile constrained intended-change contract -> execute -> review
  surfaced intended-change cases against old baseline -> explicitly confirm new baseline
  promotion as a separate action.
- Optional live-validation lane -> run a MuJoCo-enabled proof path -> confirm the same
  surfaced-case contract and verdict semantics as the deterministic fixture lane.
