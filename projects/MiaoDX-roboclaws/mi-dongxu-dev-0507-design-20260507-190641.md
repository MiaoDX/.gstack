## Problem Statement

Roboclaws needs a MolmoSpaces manipulation demo that makes robot-agent developers think: "Wow, it actually works. Let's test on hardware."

The audience is not policy-learning researchers chasing benchmark rank. The audience is robot-agent developers building ReAct/tool-calling loops who assume embodied tasks require a much heavier framework. The demo should show that `帮我收拾这个房间` can run through a small tool contract, produce a readable trace, and end in a credible before/after report.

## What Makes This Cool

The "whoa" moment is not a high-SOTA manipulation policy. It is seeing a normal coding-agent loop do household cleanup with so few moving parts:

- one runnable `just code::codex` or `just code::cc` command
- one small MCP tool surface
- one offline `report.html`
- one visible state delta from messy room to cleaner room
- one honest provenance trail that says what was real, API-backed semantic behavior, scripted behavior, or shimmed behavior

The demo wins when a robot-agent developer can inspect the report and believe the control loop is portable enough to try against hardware.

## Constraints

- AI2-THOR remains the stable baseline and regression backend.
- MolmoSpaces/MuJoCo is the next manipulation substrate, but local API facts must drive implementation.
- The first public proof should be `帮我收拾这个房间`, not a separate toy pick/place milestone.
- Manipulation may use MolmoSpaces APIs as API-backed semantic manipulation for v0.
- The report must not make semantic/API-backed behavior look like low-level real manipulation.
- Navigation can use `goto`; the navigation stack comes inside the demo path.
- Add extra abilities only when the cleanup loop exposes a real gap.
- Do not build a broad backend-neutral simulator abstraction before MolmoSpaces API facts are proven locally.
- Do not claim local-only MolmoSpaces validation from cloud-only evidence.

## Premises

1. The target user is a robot-agent developer building ReAct/tool-calling loops, not a policy-learning researcher.
2. The first proof goes straight at `帮我收拾这个房间`, using MolmoSpaces APIs as API-backed semantic manipulation where needed.
3. Public proof means one runnable `just code::codex` / `just code::cc` command plus one offline `report.html`.
4. Hardware credibility depends on provenance: the report must make `real`, `api_semantic`, `scripted`, and `shim` primitives impossible to confuse.
5. `goto` is acceptable for the demo. Navigation should be integrated into the tool loop, and additional abilities should be added only when needed.
6. A backend-neutral simulator abstraction is premature until MolmoSpaces API facts are proven locally.

## Landscape Notes

Current public MolmoSpaces and MolmoBot material is mostly shaped around scalable datasets, MuJoCo benchmark tasks, policies, and sim-to-real manipulation. That is not the Roboclaws wedge.

Roboclaws should zig: make the audited agent loop the product. The artifact should show the tiny sequence of observations, object inventory, navigation, semantic manipulation, scoring, and report generation that makes a robot-agent developer think the same loop could be connected to hardware.

## Approaches Considered

### Approach A: Cleanup-First Semantic Manipulation

Build the `帮我收拾这个房间` demo first using MolmoSpaces scene/object APIs plus API-backed semantic `pick`, `place`, `open`, and `close` behavior where needed. Keep all primitives labeled in trace and report.

- Effort: M
- Risk: Med
- Pros:
  - Produces the exact "this works with so few things" artifact fastest.
  - Forces the MCP surface to match the real user-facing task, not a toy task.
  - Reuses existing `just code::cc/codex`, trace, `run_result.json`, and report patterns.
- Cons:
  - If MolmoSpaces APIs are awkward, failures happen inside the flagship demo path.
  - Requires discipline in the report so semantic manipulation is not oversold.
- Reuses:
  - `examples/coding_agent_nav_server.py`
  - `roboclaws/mcp/server.py`
  - `roboclaws/core/run_artifacts.py`
  - `scripts/render_autonomous_replay.py`
  - existing `just code::*` workflow shape

### Approach B: Capability Matrix First

Run a narrow local script to enumerate MolmoSpaces scene load, inventory, frames, state readback, and semantic manipulation APIs before touching MCP/demo code.

- Effort: S/M
- Risk: Low
- Pros:
  - De-risks API assumptions before writing product-facing code.
  - Produces clean engineering evidence for a later GSD handoff.
- Cons:
  - Delays the actual "wow" artifact.
  - Can drift into checklist work that does not prove the user-facing loop.
- Reuses:
  - existing regression and artifact conventions, but less of the actual demo path

### Approach C: Hardware-Shaped Report Contract First

Build the cleanup demo around a hardware-portability contract: every tool call records command intent, simulator implementation, provenance, state delta, and hardware analogue.

- Effort: L
- Risk: Med
- Pros:
  - Makes the demo more credible to robot-agent developers inspecting the trace.
  - Sets up future hardware experiments without pretending sim APIs are hardware.
- Cons:
  - More design work before the first runnable demo.
  - Could overbuild the report before the MolmoSpaces API shape is known.
- Reuses:
  - existing report and trace code, but with a larger schema expansion

## Recommended Approach

Choose **Approach A: Cleanup-First Semantic Manipulation**.

This best matches the stated goal: a developer sees `帮我收拾这个房间`, opens `report.html`, inspects the tiny tool loop, and thinks "let's test this on hardware."

The implementation should still borrow one idea from Approach C: provenance is not optional. Every manipulation primitive in `trace.jsonl`, `run_result.json`, and `report.html` should carry a provenance enum such as:

```json
{
  "primitive": "place",
  "provenance": "api_semantic",
  "implementation": "molmospaces_task_api",
  "hardware_analogue": "arm_place_skill",
  "state_delta": {
    "object_id": "cup_001",
    "from": "floor",
    "to": "table_003"
  }
}
```

Suggested public command shape:

```bash
just code::codex molmo-cleanup-demo
just code::cc molmo-cleanup-demo
```

Both should run the same scenario and produce a run directory containing:

- `scenario.json`
- `trace.jsonl`
- `run_result.json`
- `report.html`
- before/after snapshots
- optional replay or frame sequence

## Success Criteria

1. A local user can run one command and trigger the MolmoSpaces cleanup demo.
2. The demo prompt is `帮我收拾这个房间`.
3. The run produces an offline `report.html`.
4. `report.html` shows initial room, final room, restored/missed object table, tool trace, score, and primitive provenance.
5. At least 3 of 5 seeded misplaced objects end on valid receptacles for the easy seed.
6. `trace.jsonl` records every tool call, result status, error reason, and primitive provenance.
7. `run_result.json` records backend, scenario id, seed, final status, score summary, artifact paths, and primitive provenance summary.
8. No private scoring manifest is exposed to the agent through MCP or copied into prompt context.
9. Failures are honest: impossible actions, stale references, missing APIs, and timeouts produce explicit terminal states.
10. The README can truthfully say the demo uses MolmoSpaces API-backed semantic manipulation where that is the implementation path.

- A good sample `report.html` should be linked from README/GitHub Pages after local validation.
- No package release is needed for this design.

CI/CD:

- CI should cover fake-backend schema, scenario parsing, scoring, privacy boundaries, report rendering, and trace/run-result compatibility.
- Real MolmoSpaces/MuJoCo validation stays local-dev until the runtime can be made CI-safe.
