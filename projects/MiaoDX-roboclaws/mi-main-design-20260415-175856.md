## Problem Statement

Roboclaws is a Python framework where multiple VLM agents control simulated robots in AI2-THOR. The goal is to show AI researchers and robot developers that making robots more agentic is surprisingly easy with small effort. The current codebase has territory and coverage games, but nothing that demonstrates *emergent cooperation* — the moment where agents with different information and capabilities must work together to succeed.

## What Makes This Cool

The "whoa" moment is watching three robots with asymmetric roles — one can see the goal, one can reach the tool, one can unblock the path — figure out how to cooperate without explicit hand-coded coordination. This turns a simulation demo into evidence that VLMs can drive social behavior in embodied agents.

## Constraints

- Must run in under 10 minutes for a new user.
- Must produce a shareable GIF/visual output.
- Must reuse existing engine, VLM, visualizer, replay, and reporter infrastructure.
- Cloud sessions cannot run real AI2-THOR or real VLMs; local validation is required.
- OpenClaw Gateway integration exists in `openclaw/bridge.py` but is partial; not a near-term dependency.

## Premises

1. The core value prop is "agentic robots are easier than people think." The deliverable must be something an AI researcher or robot developer can run in <10 minutes and immediately see emergent behavior.
2. The 10x vision is live OpenClaw Gateway integration, but the near-term path is a polished, shareable demo (paper/blog + new game mode).
3. Multi-agent sim robot coordination is a verified gap — first-mover advantage exists, but only if we ship something visible.
4. Distribution matters as much as the code. A great repo with no GIFs and no paper/blog post is invisible to researchers.

Build exactly one new example — `examples/emergence_demo.py` — where 3 robots with asymmetric info/capabilities must cooperate to retrieve a target. Ship it with a generated GIF, a short arXiv-style note or blog post, and a one-command README.

- Effort: M
- Risk: Low

- Cons: Not a reusable framework. One scenario only.

### Approach B: Emergence Game Mode + Paper
Add a new `roboclaws/games/emergence.py` game mode with configurable asymmetric roles, partial observability, and a cooperation metric. Run a small systematic study (3 seeds x 3 conditions) and write a short research note.

- Effort: L
- Risk: Med

- Cons: 2-3x the code. Needs careful metric design. Longer to ship.
- Reuses: Existing game framework, replay/reporter infrastructure.

### Approach C: Live OpenClaw Gateway Demo
Wire each robot to an independent OpenClaw Gateway agent with memory. Run a live demo where agents message each other and coordinate in real time.

- Effort: XL

- Pros: This IS the 10x vision. No one in the ecosystem has it.
- Cons: OpenClaw bridge is partial. Gateway setup is heavy. Could take weeks and still fail to demo cleanly.
- Reuses: openclaw/bridge.py

## Open Questions

- Which specific AI2-THOR scene and object combination creates the cleanest asymmetric cooperation scenario?
- What is the right balance between GPT-4o-mini (cheap, fast dev) and Claude Sonnet (stronger reasoning, better demo) for this scenario?

**Resolved assumptions:**
- Agents will **not** broadcast messages in v1; coordination emerges purely from observation and action. Broadcast can be added in a later iteration.
- Asymmetric capabilities will be enforced **at the prompt level** (role descriptions tell each VLM what it can/cannot do) rather than engine-level action filtering. This keeps the diff small and avoids engine changes.
- Private information (e.g., one agent sees the target location) will be implemented via **per-agent prompt text**, not per-agent map variants. For example, Agent C's prompt explicitly states: "You know the target is in the kitchen." This avoids visualizer changes.

**Proposed scenario (subject to feasibility pre-check):**
- Scene: `FloorPlan201` (living room, clear layout, multi-agent safe).
- Target: `RemoteControl` on the kitchen counter.
- Obstacle: A closed door between the living room and kitchen.
- Roles:
  - **Agent A (Scout):** Prompt says "You cannot pick up objects. You know the RemoteControl is on the kitchen counter." Required action: lead/guide others toward the kitchen.
  - **Agent B (Opener):** Prompt says "You can open doors but you cannot pick up the RemoteControl." Required action: open the door to the kitchen.
  - **Agent C (Retriever):** Prompt says "You can pick up objects. You do not know where the RemoteControl is." Required action: pick up the RemoteControl once reachable.

**Feasibility risks:**
- AI2-THOR multi-agent object pickup can be buggy for some object types. A 10-minute pre-check script should verify `PickupObject` on `RemoteControl` with `agentCount=3` before building the VLM loop.
- If prompt-level enforcement leaks (especially with GPT-4o-mini), a lightweight engine-level filter in `engine.py` can be added as a Day-2 patch.
- VLM outputs are non-deterministic even with a fixed random seed. The seed controls AI2-THOR initialization and local randomness; VLM temperature should be pinned to 0.0. The "2 of 3" success criterion is a statistical target, not a deterministic guarantee.

## Success Criteria

1. `python examples/emergence_demo.py --agents 3` runs end-to-end and produces a GIF.
2. The demo ships with a hardcoded seed (e.g., `--seed 42`) known to produce cooperative behavior in local validation. Local validation runs the seed at least 3 times and requires success in at least 2 runs.
3. On failure seeds, the demo terminates within the step budget and still produces a replay/GIF showing the failure mode.
4. A short writeup (primary format: blog post; optional: arXiv note) is published and linked from the README.
5. The example is validated locally with real VLM + AI2-THOR (per CLAUDE.md cloud/local split).

**Emergence metric:** Cooperation is defined as all three agents contributing at least one required action before the target is retrieved.
